# ArcVellum v0.9.0 Desktop Acceptance

This checklist covers the visual and installer checks that cannot be substituted by a source-level test run.

## Before Testing

1. Build the release with `packaging/build_desktop.ps1` from a normal Windows desktop environment. Do not substitute a direct `tauri build`: the release script rebuilds the frozen Python sidecar before packaging.
2. Ensure the build machine has the declared Python dependencies and the existing `TAURI_SIGNING_PRIVATE_KEY`; retain the configured updater public key.
3. Start with a new Studio profile or clear only the local ArcVellum window-layout preferences.
4. Open a project with at least one formal chapter, one pending task, and several library nodes.

## Spatial Workbench

Test at 1280x720, 1440x900, 1920x1080, 2560x1440, and 390x844:

1. Enter the v3 narrative stage, pan in all directions, zoom under the pointer, use Fit, and focus a scene node.
2. Confirm foreground, middle, and distant layers are visible, while the layout remains a narrative field rather than a literal map.
3. Open a node, progress, reader, decisions, rules, and delivery window. Drag and resize each window; its dimensions and scroll region must remain stable after release.
4. Open more than twelve windows. Older windows must move into the compact minimized rail and restore without losing their content or position.
5. Switch themes, motion, depth, and render quality. The scene, nodes, window material, and backgrounds must change coherently.
6. Turn on reduced motion and verify that the same task, decision, reader, rule, health, and delivery actions remain reachable.

## Runtime and Delivery

1. Start an ordinary task and verify that the progress instrument receives status updates without exposing internal reasoning.
2. Complete an export-and-release route. The delivery panel must update through its stream, show generated files, and download through the authenticated local endpoint.
3. Disconnect or leave OpenCode unconfigured. The interface must explain the condition without claiming that a task finished.
4. Lose and restore the local service connection. Existing project state, current task, and reader position must recover.

## Installer

1. Verify Tauri cold start does not flash a terminal window and reaches the loading sequence.
2. Test a clean NSIS install, in-place upgrade, uninstall, and restart.
3. Confirm the default project location, directory chooser, API session, update manifest, and diagnostic export all work after installation.
4. Confirm the packaged sidecar reports the same `0.9.0` application version through the local health endpoint before testing project work.

Record screenshots for every viewport and attach any visual overlap, blank canvas, unreadable contrast, or task-state mismatch to the release issue before publishing.
