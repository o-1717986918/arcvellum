# Studio 功能完备度审查（v0.2.1）

本审查基于 `src/literary_engineering_studio/`、嵌入式 `literary_engineering_studio_engine`、前端六个一级页面、17 项 Studio 自动测试和桌面/移动浏览器验收，不以 README 声明代替实现证据。

## 结论

Studio 已经是可运行的独立工程化 MVP，但还不是无需维护者介入的完整桌面创作产品。

- 文学工程状态机、任务包、隔离写回、正式门禁和项目只读模型较成熟。
- 项目创建、方向追加、任务准备与运行、人工选择、作品档案、文风挂载、交付检查已经形成前端主链。
- 产品级项目生命周期、Worker 恢复、写回差异确认和完整文风学习仍不完整。

## 仍不完整的产品能力

### 1. 项目生命周期

当前只能创建、打开和切换。缺少重命名、归档、从最近项目移除、复制、备份、迁移和失效路径修复。浏览器构建也只能填写目录文本，尚无桌面原生文件夹选择器。

### 2. Worker 持久性

`JobStore` 保存执行记录，但任务仍由进程内线程运行。服务重启后不能自动恢复正在执行的任务，也没有停止、重试、续跑、每项目任务锁、幂等键和并发上限。

### 3. 写回前确认

沙箱能够限制输出范围，但高影响任务尚缺统一的“写回差异预览”。Canon、人物事实、正式正文与发布候选应在导入前展示新增、修改、删除和影响范围，再由用户确认。

### 4. 文风学习客户端

前端能够检查和挂载 Style Skill，但作家项目创建、作品导入、证据分析、提示词生成、回译/扩写评估和正式晋升仍需通过状态机任务推进，没有形成独立的前端向导。

### 5. 结构化编辑

作品档案提供展示字段和备注层编辑，不允许用户直接破坏正式文件，这是正确边界；但项目 Brief、目标规模、卷章结构、主要角色、世界约束等高频修改尚缺“生成候选 -> 预览差异 -> 审查 -> 应用”的结构化表单。

### 6. 交付深度

v0.2.1 已能展示门禁和下载现有交付文件，但还缺导出版本对比、发布备注、封面与版式预览、直接打开输出目录、历史回滚和多格式批次管理。

### 7. 产品级可靠性

当前 17 项测试覆盖核心正向路径和关键安全边界，但缺少真实浏览器交互测试、长任务中断恢复、并发冲突、大型项目性能、SSE 断线重连压力和 Windows/macOS/Linux 适配矩阵。

### 8. 本地服务边界

服务默认监听 `127.0.0.1`，适合本机使用。若未来允许局域网或公网访问，必须增加身份认证、允许项目根目录列表、CSRF 防护、下载审计和更严格的进程权限隔离。

## 下一阶段优先级

1. Worker 停止、重试、恢复、项目锁和写回差异预览。
2. 项目归档、复制、备份和失效路径修复。
3. 文风学习完整前端向导。
4. 结构化人物、世界观和大纲候选编辑。
5. 交付版本历史、版式预览与回滚。
6. 端到端浏览器测试与大型项目性能测试。

Codex CLI 适配不在本轮优先级内；上述工作均可先围绕宿主 Agent 与 Claude Code 完成。
