# ArcVellum v0.7.1-v0.8 创作控制与叙事星仪计划

## 1. 文档状态

- 基线版本：`ArcVellum v0.7.0`
- 计划状态：已立项，`v0.7.1` 安装版连接热修正在实施
- 目标平台：Windows 10/11 桌面客户端
- 计划范围：安装版可靠性、创作品质配置、Narrative Orrery、贯通式启动体验、帮助中心、作品与应用详情、合规说明
- 架构前提：Tauri 2、Vue 3、FastAPI 本地服务、内嵌文学工程内核、捆绑 OpenCode Agent Runner
- 评审依据：`v0.7.0` 源码、安装目录实测、Tauri sidecar 进程、浏览器 CORS 契约、现有 Reader、Advisor 与 Narrative Projection

本计划延续 `docs/roadmap/arcvellum-v0.5.1-v0.7-reader-advisor-observatory-plan.md`。已经实现且通过验证的完整阅读器、顾问人格、主动消息、SSE、项目库和 Narrative Projection 不重新建设。

## 2. 产品目标

下一阶段要解决两个不同层次的问题：

1. 让安装后的普通用户稳定进入应用，并能理解、诊断和恢复本地服务故障；
2. 让 ArcVellum 从“带故事图的创作工作台”进化为“以动态叙事观测为视觉核心、以创作品质配置为用户控制面的文学创作产品”。

目标体验：

- 安装后无需开发环境即可稳定进入正式页面；
- 用户能用文学语言管理标点、表达习惯和 Lint 强度，不需要理解正则或 CLI 参数；
- 创作规则变化可追溯，不会无声改变已经通过的审查结论；
- 动态故事脉络是首页的常态主体，而不是 Dashboard 中的一张流程图；
- 启动动画与正式故事画布是同一视觉过程；
- “使用帮助”能够解释当前状态并提供恢复动作；
- 作品状况、应用信息、开发者、官网、协议和第三方组件都有正式入口；
- 内容创作保持开放，工程权限、数据安全和公开发布责任保持明确。

## 3. 当前实现结论

### 3.1 可以复用

- Narrative Projection 已能投影章节、场景、人物、分支、Canon 与 Promise/Payoff；
- Reader Manifest、正式正文去重、增量阅读和阅读位置已经建立；
- Advisor 已具备人格、主动消息和只读项目上下文；
- Tauri 能启动冻结 sidecar，并向前端注入随机端口和桌面令牌；
- FastAPI 已有桌面会话、CORS、健康检查和诊断导出基础；
- 创作引擎已有 Anti-AI Style、标点标准、AgentReview 和 Promotion Gate；
- Vue 路由、设置页、SSE Store 和桌面更新器可以继续使用。

### 3.2 已确认缺口

1. 安装版 WebView 跨源请求使用 `credentials: include`，后端却返回 `allow_credentials=False`，浏览器会将有效响应拦截为 `Failed to fetch`。
2. 发布验证只检查冻结 sidecar 的 HTTP `/health`，没有覆盖“已安装 Tauri WebView + CORS + 桌面令牌”的完整链路。
3. 创作限制分散在 Python 常量、规则函数和 Prompt 中，用户无法建立项目级创作策略。
4. 破折号规则的提示语声称使用约 2% 密度，实际检测同时采用绝对数量阈值，显示语义与算法不一致。
5. 当前 StoryTrace 是固定横向 SVG 节点图，数据正确，但视觉隐喻和页面层级不符合“常态视觉主体”的目标。
6. 启动页完成后被卸载，再显示正式界面，形成明显的“启动页切换到应用”断点。
7. 左下角“使用帮助”没有交互行为。
8. 设置页只有简要“关于”，缺少独立的作品详情、应用详情、帮助中心和协议入口。
9. OpenCode、本地 Agent、第三方模型和 ArcVellum 自身的使用边界尚未用普通用户能理解的方式说明。

## 4. 硬边界

### 4.1 用户可以管理的创作策略

- 引号样式与中文标点偏好；
- 句长、段长、逗号链和短句密度；
- 破折号、省略号、显式转折词与对照句式；
- 器官反应、空洞比喻、解释性心理描写和口号式收尾；
- 每条规则的关闭、提醒、阻断三级强度；
- 项目预设、场景临时例外和规则说明；
- 文风偏好、禁用词与例外样例。

### 4.2 用户不能关闭的工程门禁

- CLI 状态机、来源证明和精确候选哈希；
- AgentReview、Promotion、Canon、State 与正式导出门禁；
- Schema、路径、写回范围和正式正文清洗；
- 禁止 debug waiver、unreview 和伪造正式产物；
- 正文只能由主 Agent 完成；
- 公开发布前的完整性与版权责任确认。

创作偏好属于作品政策，工程门禁属于产品完整性。前端不得把两者混成一个“严格度”总开关。

## 5. 实施阶段

### 5.1 `v0.7.1`：安装版连接热修

目标：修复安装后无法进入正式页面的问题，并补足生产路径验证。

任务：

1. 允许受信任的 Tauri 来源携带桌面会话凭据；
2. 保留随机 Bearer 启动令牌和受限 Origin 白名单；
3. 将 `Failed to fetch`、`Cannot fetch`、`NetworkError` 转为可操作的中文提示；
4. 增加 Tauri Origin 的 CORS 预检回归测试；
5. 增加安装后启动、桌面会话、Bootstrap、项目列表的端到端烟测；
6. 诊断导出包含 sidecar 启动状态、监听端口、WebView Origin 和最近启动失败原因；
7. 构建可覆盖安装的 `0.7.1` NSIS 包，并在非系统盘安装目录验证。

完成条件：

- 安装版冷启动后自动进入作品页或总控页；
- `/desktop/session`、`/application/bootstrap`、`/projects` 均通过真实 WebView 请求；
- sidecar 失败时不显示英文 `Failed to fetch`；
- 重试能重新执行桌面会话和 Bootstrap；
- 开发版 Vite 代理与浏览器调试模式不回归。

### 5.2 `v0.8.0-A`：创作品质配置

目标：将硬编码文学偏好升级为可版本化、可解释、可预览的项目策略。

新增 `Creative Quality Profile v1`，至少包含：

- 配置 ID、版本、名称、继承的系统预设；
- 标点与引号规范；
- 表达模式规则及关闭、提醒、阻断级别；
- 以“每 100 个叙事单元”或“每万中文内容字符”表示的阈值；
- 文风挂载与规则冲突时的优先级；
- 场景级例外、理由、有效范围和过期条件；
- Profile digest、创建时间、修改来源和适用起点。

前端分为两层：

- 基础模式：预设、6 至 8 个文学含义明确的控制项、样文预览；
- 高级模式：逐条规则、阈值、作用域、例外和阻断级别。

禁止向普通用户展示正则表达式、Python 常量名和 CLI 内部字段。

链路要求：

1. Compose、Generate、Review、Revise 和 Promote 读取同一 Profile；
2. Prompt package 写入 Profile digest；
3. Review 报告记录实际使用的版本；
4. 修改 Profile 默认只影响未来候选稿；
5. 旧稿必须重新 Review，不能因降低阈值自动变为通过；
6. Release Manifest 固化交付时的 Profile 快照；
7. Profile 冲突时，文风控制表达，工程规则控制完整性，场景例外必须显式登记。

完成条件：

- 用户能在不理解 Lint 实现的情况下预览一段文本会触发哪些规则；
- 同一候选稿在相同 Profile 下得到确定性的静态检测结果；
- 生成约束、AgentReview 和 Promotion 不再使用不同版本的规则；
- 破折号等密度提示与实际算法完全一致。

### 5.3 `v0.8.0-B`：Narrative Orrery

目标：把动态叙事星仪做成 ArcVellum 的唯一重点视觉和首页常态主体。

布局：

- 56 至 64 像素窄导航栏；
- 中央 70% 至 80% 全幅叙事画布；
- 右侧按需出现的详情与决策抽屉；
- 底部卷、章、场景时间轴；
- 任务、健康、审批和配置进入边栏或抽屉；
- 阅读器继续保留独立、安静的长文页面。

视觉语义：

- 正式故事线：持续生长的 Canon 长卷；
- 候选分支：从主线分出的细线，落选后淡出但可追溯；
- 角色：穿过场景的行动轨迹，不使用装饰性头像；
- Promise/Payoff：跨场景闭合弧；
- Canon 写回：锚点扩散后稳定；
- 当前任务：沿轨迹移动到目标场景的状态脉冲；
- 正文字数：长卷厚度、纹理或轮廓密度；
- Review 债务：局部琥珀色干扰，不使用全屏警报。

技术路线：

- Vue 负责导航、抽屉、文字和无障碍界面；
- PixiJS/WebGL 负责叙事画布；
- Narrative Projection 继续作为唯一事实来源；
- 新增 Projection Delta、事件序号、聚合层级和焦点邻域；
- 1000 场景项目默认按卷章聚合，只有焦点邻域展开；
- 图形不可用时自动切换为现有列表投影；
- 支持键盘、屏幕阅读器和 `prefers-reduced-motion`。

动效必须由真实事件触发：

- 创建场景时轮廓生长；
- 生成分支时线路分叉；
- 选择后非正式分支淡出；
- 正文生成后长卷增加厚度；
- 晋升后候选线并入正式线；
- Canon 应用后关联锚点点亮；
- 顾问建议获批后，指令包进入目标节点。

完成条件：

- 每个可视对象都能追溯到正式事实或运行事件；
- 没有与状态无关的持续粒子和随机闪光；
- 390x844、1024x768、1440x900、1920x1080 均可操作；
- 帧率下降不会阻塞创作、审批、阅读和导出；
- 视觉主体明显强于其他模块，但长期阅读区域保持克制。

### 5.4 `v0.8.0-C`：贯通式启动体验

目标：让启动动画成为叙事星仪的第一状态，而不是一次性 Splash。

时序：

1. 校准线出现；
2. 根据真实 Bootstrap Step 绘制故事丝线；
3. 品牌从轨迹交汇处显现；
4. 镜头推进到当前作品和当前场景；
5. 导航、时间轴和必要抽屉进入；
6. 同一个 Canvas 留在正式页面继续工作。

约束：

- 首次完整动画 1.8 至 2.4 秒；
- 后续启动压缩到 0.45 至 0.7 秒；
- 慢启动显示真实阶段，不显示伪造百分比；
- 600 毫秒后允许跳过视觉动画，但不能跳过服务准备；
- Reduced Motion 使用静态品牌和真实状态文本；
- 启动失败时画布停在故障位置，并提供重试、诊断和退出。

### 5.5 `v0.8.0-D`：帮助、详情与说明

“使用帮助”改为上下文帮助抽屉，并连接完整 `/help` 页面。

帮助中心至少包含：

- 第一次使用；
- 当前步骤在做什么；
- 为什么需要审批或被门禁阻止；
- 模型、Agent 与本地数据的关系；
- 协作、监督和全自动模式；
- 阅读、交付、更新与备份；
- 常见启动和连接故障；
- 诊断导出与问题反馈；
- 快捷键和无障碍操作。

详情拆为两类：

- 作品详情：目标字数、正式字数、卷章场景、挂载文风、创作策略版本、健康、备份和活跃任务；
- 应用详情：版本、构建、更新渠道、开发者、仓库、官网、支持方式、数据位置、隐私、协议和第三方许可证。

官网或支持地址为空时不展示伪链接。外部地址由系统浏览器打开并经过允许列表校验。

### 5.6 `v0.8.0-E`：使用协议与开放创作边界

原则：不在本地创作内核中增加宽泛、不可解释的文学内容过滤，但必须说明不同运行层的规则来源。

必须区分：

- OpenCode 开源执行器的 MIT 许可；
- OpenCode 托管服务的服务条款；
- 第三方模型 Provider 的内容政策、拒答和数据保留；
- ArcVellum 本地文件、Agent 权限和发布流程；
- 用户对素材来源、版权、真实人物、未成年人和公开传播的责任。

最低产品能力：

- 首次启用 Agent 时展示数据流向；
- 全自动模式和最终发布前进行明确确认；
- 导入作品或作家语料时记录权利来源声明；
- 提供隐私说明、使用协议和第三方许可证；
- 内容开放不能削弱 Shell、文件、任务沙箱和写回边界；
- 为未来公共服务、商店分发和未成年人用户预留可插拔 Policy Provider，不在本地 Beta 强制启用。

## 6. 视觉设计流程

在全量重构前完成两个可交互原型：

1. Midnight Moss 叙事星仪：深苔画布、玉石轨迹、黄铜记忆锚点；
2. Mineral White 光纸仪器：明亮工作面、墨色长卷、低饱和状态色。

原型必须使用真实 Narrative Projection Fixture，不使用虚构 Dashboard 数字。分别在桌面和窄屏截图、录制 20 至 30 秒状态动画并进行自我批评。最终只保留一个重点视觉，删除不能表达真实状态的装饰。

Image2 只用于情绪板、背景纹理和概念参考。功能图标继续使用 Lucide 或代码原生矢量；故事轨迹、节点和动画由代码渲染，不能用一张静态大图冒充动态产品能力。

## 7. 测试与发布门槛

### 7.1 自动化

- Python 全量回归；
- Vue/Vitest、TypeScript 和生产构建；
- Rust `cargo check --locked`；
- Tauri Origin CORS 与桌面会话；
- Creative Quality Profile schema、迁移、digest 和重审行为；
- Projection Delta 顺序、重复事件和断线恢复；
- 1000 场景聚合性能；
- Help、About、协议和外链允许列表；
- 凭证扫描、许可证清单和 updater manifest。

### 7.2 安装版端到端

每个候选版本必须在真实安装目录验证：

1. 干净安装；
2. 覆盖安装；
3. 非系统盘安装；
4. 中文用户名；
5. 无网络启动；
6. sidecar 被阻止或端口占用；
7. WebView 发起桌面会话、Bootstrap、项目列表和 SSE；
8. 关闭主程序后 sidecar 自动退出；
9. 卸载后用户作品不被删除；
10. 更新器下载、签名和回滚提示。

仅用 `Invoke-WebRequest` 检查 `/health` 不再视为桌面客户端通过。

## 8. 推荐实施顺序

1. 完成并发布 `v0.7.1` 安装版连接热修；
2. 统一创作品质规则的检测语义和 Profile 契约；
3. 完成两套 Narrative Orrery 交互原型并确定视觉方向；
4. 扩展 Projection Delta 与聚合接口；
5. 重构首页壳与持久画布；
6. 接入贯通式启动动画；
7. 完成创作规则管理页、样文预览和规则版本链路；
8. 完成帮助中心、作品详情、应用详情和协议；
9. 做性能、无障碍、安装版端到端和发布验证；
10. 发布 `v0.8.0`，保留 `v0.7.1` 作为可回滚稳定版本。

## 9. 完成定义

本计划只有在以下条件同时满足时才算完成：

- 安装版不存在 `Cannot fetch` 阻塞；
- 创作规则可以由用户理解、预览和管理，但不能关闭工程门禁；
- 生成、Review、Revise、Promote 和 Release 使用同一规则版本；
- 叙事星仪成为首页常态主体，并由真实项目状态驱动；
- 启动动画与正式画布连续；
- 使用帮助、作品详情、应用详情、隐私和协议均可访问；
- 大型项目、低性能设备和 Reduced Motion 下仍可用；
- 安装版端到端验证、全量测试、发布签名和更新清单全部通过。
